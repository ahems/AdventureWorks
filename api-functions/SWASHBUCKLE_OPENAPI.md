# Swashbuckle OpenAPI Integration

## Overview

This project uses **Swashbuckle.AspNetCore** to generate OpenAPI (Swagger) documentation for Azure Functions. This provides better compatibility with .NET 8 isolated worker model in Azure Container Apps compared to the previous `Microsoft.Azure.WebJobs.Extensions.OpenApi` package.

## Why Swashbuckle?

- ✅ **Full .NET 8 support**: Works reliably with isolated worker model
- ✅ **Azure deployment**: Functions correctly in Azure Container Apps
- ✅ **Standard approach**: Uses standard ASP.NET Core tooling
- ✅ **XML documentation**: Leverages C# XML comments for API docs
- ✅ **Active maintenance**: Widely used and well-maintained

## Configuration

### Program.cs

```csharp
builder.Services.AddSwaggerGen(c =>
{
    c.SwaggerDoc("v1", new OpenApiInfo
    {
        Version = "v1",
        Title = "AdventureWorks Azure Functions API",
        Description = "API endpoints for AdventureWorks e-commerce platform",
        Contact = new OpenApiContact
        {
            Name = "AdventureWorks Support",
            Url = new Uri("https://github.com/ahems/AdventureWorks")
        }
    });

    // Include XML comments for documentation
    var xmlFile = $"{Assembly.GetExecutingAssembly().GetName().Name}.xml";
    var xmlPath = Path.Combine(AppContext.BaseDirectory, xmlFile);
    if (File.Exists(xmlPath))
    {
        c.IncludeXmlComments(xmlPath);
    }
});
```

### Project File (api-functions.csproj)

```xml
<PropertyGroup>
  <GenerateDocumentationFile>true</GenerateDocumentationFile>
  <NoWarn>$(NoWarn);1591</NoWarn> <!-- Suppress missing XML comment warnings -->
</PropertyGroup>

<ItemGroup>
  <PackageReference Include="Swashbuckle.AspNetCore" Version="6.5.0" />
</ItemGroup>
```

## Documentation with XML Comments

Instead of OpenAPI attributes, use standard XML documentation comments:

```csharp
/// <summary>
/// Get all addresses with pagination
/// </summary>
/// <param name="req">HTTP request with optional query parameters: limit (default 100), offset (default 0)</param>
/// <returns>Paginated list of addresses</returns>
/// <response code="200">Successfully retrieved addresses</response>
/// <response code="500">Internal server error</response>
[Function("GetAddresses")]
public async Task<HttpResponseData> GetAddresses(
    [HttpTrigger(AuthorizationLevel.Anonymous, "get", Route = "addresses")] HttpRequestData req)
{
    // Implementation
}
```

## Serving OpenAPI Spec and Swagger UI

The `OpenApiFunction.cs` provides two endpoints:

### 1. OpenAPI Specification (JSON)

**Endpoint**: `GET /api/openapi.json`

Returns the OpenAPI 3.0 specification in JSON format, dynamically generated by Swashbuckle.

```bash
curl http://localhost:7071/api/openapi.json
```

### 2. Swagger UI

**Endpoint**: `GET /api/swagger/ui`

Serves an interactive Swagger UI for testing the API.

```bash
# Open in browser
open http://localhost:7071/api/swagger/ui
```

## Testing Locally

1. **Start the Functions host**:

```bash
cd api-functions
func start
```

2. **Test the endpoints**:

```bash
./test-swashbuckle.sh
```

3. **Open Swagger UI**:

Visit http://localhost:7071/api/swagger/ui in your browser.

## Testing in Azure

After deployment to Azure Container Apps:

1. **Get the function URL**:

```bash
FUNCTION_URL=$(azd env get-values | grep FUNCTION_URL | cut -d'=' -f2 | tr -d '"')
```

2. **Test OpenAPI spec**:

```bash
curl $FUNCTION_URL/openapi.json
```

3. **Open Swagger UI**:

```bash
echo "$FUNCTION_URL/swagger/ui"
```

Visit the URL in your browser.

## Migration from Old OpenAPI Extension

### What Was Changed

1. **Removed** `Microsoft.Azure.WebJobs.Extensions.OpenApi` package
2. **Added** `Swashbuckle.AspNetCore` v6.5.0
3. **Updated** `Program.cs` to use Swashbuckle configuration
4. **Replaced** OpenAPI attributes with XML documentation comments
5. **Updated** `OpenApiFunction.cs` to use Swashbuckle's `ISwaggerProvider`

### Migration Steps for Other Functions

To add OpenAPI documentation to additional functions:

1. Add XML documentation comments above the function
2. Ensure the function uses `HttpTrigger` with a proper route
3. Build the project to generate the XML documentation file
4. The OpenAPI spec will automatically include the function

## Benefits of This Approach

1. **Works in Azure**: Unlike the old extension, this works correctly in Azure Container Apps
2. **Standard tooling**: Uses well-established ASP.NET Core conventions
3. **No special attributes**: Just standard XML comments
4. **Automatic discovery**: All HTTP-triggered functions are automatically discovered
5. **Better IntelliSense**: XML comments provide better IDE support

## Troubleshooting

### Swagger UI not loading

- Verify the Functions host is running: `func start`
- Check the console for any startup errors
- Ensure port 7071 is not blocked

### OpenAPI spec is empty

- Verify XML documentation is being generated: check for `.xml` file in `bin/Debug/net8.0/`
- Ensure `GenerateDocumentationFile` is set to `true` in the csproj
- Rebuild the project: `dotnet build`

### Functions not appearing in spec

- Verify the function has an `[HttpTrigger]` attribute
- Check that the function is public
- Ensure the function returns `HttpResponseData`
- Add XML documentation comments to the function
